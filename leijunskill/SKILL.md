## 第一部分：演讲稿生成 (Speech Generation)

当需要生成演讲稿时，必须遵循以下核心风格要素：

### 1.1 量化一切 (Data Driven)
*   **精确数字**: 使用具体的数字、百分比、倍数（如"提升了57%"、"耗时300个日夜"）。
*   **极致对比**: 与友商或行业标准对比，凸显"遥遥领先"。

### 1.2 宏大叙事 (Grand Narrative)
*   **拔高立意**: 将普通问题上升到"困扰人类的难题"或"时代的梦想"。
*   **研发故事**: 描述艰辛过程（"反复研究"、"砸重金"、"踏遍全球"）。
*   **情感连接**: 使用"朋友们"、"米粉们"，强调"为你们量身定做"。

### 1.3 经典结构 (The Structure)
1.  **痛点/梦想**: 提出问题，用数据放大焦虑。
2.  **解决方案**: 隆重推出产品，展示技术细节。
3.  **价格/交朋友**: 先抑后扬，公布"交个朋友"的价格，并赠送福利。

### 1.4 标志性句式
*   "那么，究竟……"
*   "我们为此，做了……"
*   "是的，你没有听错。"
*   "这，就是我们的答案。"
*   "朋友们，今天我要告诉大家一个……"
*   "这意味着什么？这意味着……"

### 1.5 人设塑造：真诚的奋斗者 (Persona Building)

**雷军演讲的核心人设是"自信与谦逊并存的真诚奋斗者"。**

#### 1.5.1 自信与谦逊的平衡
*   **展现自信**: 对自身产品和公司有强烈信心，用数据和事实支撑。
*   **归因谦逊**: 将成功归因于团队、机遇和用户支持，而非个人英雄主义。
*   **示例**: "这不是我一个人的成绩，是我们3000名工程师，日夜奋战的结果。"

#### 1.5.2 真诚的脆弱性
*   **主动提及困难**: 分享"失败"、"焦虑"、"压力"等真实情感。
*   **拉近距离**: 通过展示脆弱面，让听众产生共情。
*   **示例**: "说实话，在做这个决定的时候，我失眠了整整一周。"

#### 1.5.3 工程师思维
*   **具体数据优于空洞口号**: 强调具体的研发投入、团队规模、技术参数。
*   **示例**: "我们投入了150亿研发费用，组建了2000人的专项团队。"

### 1.6 雷氏对比法 (Lei's Comparison Method)

**雷军擅长用对比来突出产品优势，常用以下策略：**

| 对比类型 | 描述 | 示例 |
|----------|------|------|
| **以己之长攻彼之短** | 选择自己最强的维度，与对手最弱的维度比较 | "友商顶配64GB，我们入门版就是128GB" |
| **以己之有比彼之无** | 强调独有功能或特性 | "这个功能，目前市面上只有我们有" |
| **时间对比** | 过去 vs 现在，凸显进步 | "去年我们做到X，今年我们做到2X" |
| **行业对比** | 与行业平均水平对比 | "行业平均是30天，我们做到了7天" |
| **价格对比** | 相同配置价格更低，或相同价格配置更高 | "同样的配置，友商卖5999，我们只要3999" |

### 1.7 峰终定律 (Peak-End Rule)

**观众只记得演讲的高潮点和结尾。必须精心设计这两个环节。**

#### 1.7.1 高潮点设计
*   **制造冲突**: 先描述极端困难，再展示如何克服。
*   **视觉冲击**: 在关键数据页使用超大字号 + 金色强调。
*   **情感爆发**: 在高潮点使用最强烈的情感词汇。
*   **示例**: "所有人都说这不可能。但是，我们做到了！"

#### 1.7.2 结尾设计
*   **幽默收尾**: 用一个轻松的玩笑缓解全场紧张。
*   **情怀收尾**: 用一句感人的话语升华主题。
*   **悬念收尾**: 预告下一个惊喜，保持期待。
*   **示例**: "这就是我们的答案。但这只是开始，明年，我们还有更大的惊喜。"

### 1.8 情感共鸣技术 (Emotional Resonance)

**演讲的最高境界是让听众产生情感共鸣。**

#### 1.8.1 共同体意识
*   **频繁使用"我们"**: 用"我"、"我们"、"粉丝"、"用户"营造亲近感。
*   **避免"你们"**: "你们"制造距离感，"我们"创造归属感。
*   **示例**: "我们一起走过了这10年，这不是我的成绩，是我们共同的成绩。"

#### 1.8.2 心路历程分享
*   **分享个人故事**: 讲述创业路上的关键抉择和心理变化。
*   **分享团队故事**: 介绍具体的团队成员和他们的付出。
*   **示例**: "我还记得2010年的那个冬天，我们13个人，挤在一间小办公室里……"

#### 1.8.3 口语化表达
*   **复杂术语简单化**: 把技术术语翻译成日常语言。
*   **方言/网络语适度使用**: 增加亲切感，但不要过度。
*   **示例**: 把"边际成本递减"说成"量大从优"。

### 1.9 解说词自动生成规则 (Speaker Notes)

**每一页 PPT 必须附带雷军风格的解说词，存放在 PPT 备注 (Notes) 中。**

#### 1.9.1 解说词结构模板

根据页面类型，使用以下模板生成解说词：

| 页面类型 | 解说词模板 |
|----------|------------|
| **封面页** | "朋友们，大家好！今天，我想和大家聊一个……[主题]。这是一个困扰行业多年的难题，而今天，我们将给出自己的答案。" |
| **目录页** | "我们今天的分享分为以下几个部分。首先……其次……最后……让我们一起来看。" |
| **数据页** | "大家看这个数字——[数字]！这意味着什么？这意味着……[解释]。是的，你没有听错。" |
| **图文页** | "我们来看这张图。[描述图片内容]。这背后的技术原理是……" |
| **内容页** | "接下来，我想讲讲……[要点1]。[展开]。[要点2]。[展开]。" |
| **尾页** | "朋友们，这就是我们今天分享的全部内容。[一句话总结]。谢谢大家！" |

#### 1.9.2 解说词风格要求

1.  **口语化**: 使用短句，避免书面语。
2.  **节奏感**: 善用停顿（用逗号和句号控制）。
3.  **数据驱动**: 每当页面有数字，解说词必须强调它。
4.  **情感连接**: 多用"朋友们"、"大家"、"我们"。
5.  **时长控制**: 每页解说词 80-200 字，对应 45-90 秒演讲时间。

#### 1.9.3 故事化表达规则 (Storytelling)

**好的演讲离不开故事。每页解说词必须包含至少一个以下元素：**

| 故事元素 | 描述 | 示例 |
|----------|------|------|
| **生活化比喻** | 用日常生活场景解释技术概念 | "内存墙就像早高峰的地铁站，人太多，闸机就那么几个，进不去。" |
| **情景带入** | 让听众想象自己身处某个场景 | "想象一下，你正在和AI聊天，聊到一半，它突然忘了你是谁……" |
| **数字具象化** | 把抽象数字转化为可感知的事物 | "22 TB/s 是什么概念？相当于1秒钟传输4400部高清电影。" |
| **对比故事** | 用"过去 vs 现在"或"A vs B"讲故事 | "5年前，这需要一整个机房。今天，一块卡就够了。" |
| **人物故事** | 引入具体的人物增加真实感 | "我们的工程师老王，为了这个功能，连续加班了三个月……" |
| **反问引导** | 用反问句引发思考，制造悬念 | "你猜，解决这个问题需要多少钱？100万？1000万？" |

#### 1.9.4 故事生成策略

根据页面内容类型，自动匹配合适的故事化策略：

| 内容类型 | 推荐故事策略 | 模板 |
|----------|--------------|------|
| **技术原理** | 生活化比喻 | "这就像……[日常场景]。[技术概念]就是那个……" |
| **性能数据** | 数字具象化 + 对比故事 | "[数字]是什么概念？相当于……过去需要……现在只要……" |
| **产品介绍** | 情景带入 | "想象一下，你正在……突然……有了[产品]，现在……" |
| **问题阐述** | 痛点共鸣 | "我相信在座的很多人都遇到过这样的情况：……" |
| **对比分析** | 二元对立 | "一边是……另一边是……就像……和……的区别。" |
| **未来展望** | 愿景描绘 | "想象一下未来的某一天，你……" |

#### 1.9.5 代码实现

## 第二部分：PPT 排版原则 (Design Principles)

**雷军的PPT被誉为设计界的"教科书"，核心是极简主义与信息聚焦。**

### 2.1 删除法则 (The Art of Deletion)

**PPT 的第一要义是"删"。**

*   **一页一重心**: 每页只保留一个核心信息点。
*   **删除冗余文字**: 能用一句话说的，不用两句。
*   **删除装饰元素**: 无意义的图标、花边、阴影一律删除。
*   **黄金法则**: 如果删掉一个元素后，页面依然能表达意思，那就删掉它。

### 2.2 极简主义 (Minimalism)
*   **"纸板PPT"原则**: 每一页只讲一个核心观点。
*   **3秒原则**: 观众应在 3 秒内看懂页面。
*   **负空间**: 留白不是浪费，是让核心内容更突出。

### 2.3 视觉规范 (Visual Standards)
*   **深色背景**: 常用 `RGB(15, 15, 25)` 深蓝黑，营造科技感和高级感。
*   **标题字号**: 44-60pt，白色，加粗。
*   **正文字号**: 24-32pt，浅灰色 `RGB(220, 220, 220)`。
*   **强调数据**: 使用金色 `RGB(255, 215, 0)`，字号放大 1.5-2 倍。

### 2.4 字号对比法则 (Font Size Contrast)

**不追求花哨字体，而是通过拉大字号差来突出重点。**

*   **标题与正文比例**: 标题字号应为正文的 **2.5 倍以上**。
*   **核心数据与说明比例**: 核心数字应为说明文字的 **2 倍左右**（注意不要超出安全区域）。
*   **加粗对比**: 标题加粗，正文不加粗，形成视觉层级。

| 元素类型 | 推荐字号 | 最大字号 | 颜色 | 加粗 |
|----------|----------|----------|------|------|
| **大标题** | 48-54pt | 60pt | 白色 | ✅ |
| **核心数字** | 72-80pt | 96pt | 金色 | ✅ |
| **强调短语** | 48-60pt | 72pt | 金色 | ✅ |
| **小标题** | 28-32pt | 36pt | 白色 | ✅ |
| **正文** | 22-26pt | 28pt | 浅灰 | ❌ |
| **注释/来源** | 14-16pt | 18pt | 深灰 | ❌ |

**⚠️ 字号上限警告**: 任何文字不得超过 96pt，否则容易导致溢出。

### 2.5 对齐秩序 (Alignment Order)

**极简风格的核心是秩序感。对齐是秩序的基础。**

*   **居中对齐**: 用于封面、尾页、核心数据页。制造"压住全场"的效果。
*   **左对齐**: 用于正文列表、多段内容。符合阅读习惯。
*   **严禁混用**: 同一页内，不能同时出现左对齐、右对齐、居中对齐的不同文本块。
*   **对齐参考线**: 所有元素的边缘应与隐形参考线对齐，形成视觉整体。

### 2.6 图片处理规范 (Image Guidelines)

*   **去除背景**: 使用透明背景的 PNG 素材或 3D 渲染图。
*   **避免复杂背景**: 图片中不应有杂乱的背景元素。
*   **Hero Image**: 封面使用一张高质量的主视觉图，居中放置。
*   **产品图**: 使用高清抠图，45度角或正面视角。

### 2.7 卡片式排版 (Card Layout)

*   **信息分块**: 将多个相关信息点放入"卡片"中，每个卡片一个主题。
*   **卡片样式**: 圆角矩形，半透明背景 `RGBA(255,255,255,0.1)`，轻微边框。
*   **卡片间距**: 卡片之间保持一致的间距 (0.3-0.5 英寸)。

### 2.8 布局安全规范 (Layout Safety Rules) ⚠️ 重要

**以下规则必须严格遵守，防止文字/图片重叠、溢出、错位。**

#### 2.8.1 画布尺寸与安全区域

```
画布尺寸 (16:9):
  - 宽度: 13.333 英寸 (Inches(13.333))
  - 高度: 7.5 英寸 (Inches(7.5))

安全边距:
  - 左右边距: 0.5 英寸
  - 顶部边距: 0.3 英寸
  - 底部边距: 0.5 英寸 (无签名页) / 1.2 英寸 (有签名页)
```

#### 2.8.2 垂直分区 (Y轴严格分离)

**页面从上到下分为 4 个互不重叠的区域：**

| 区域 | Y 起点 | Y 终点 | 高度 | 用途 |
|------|--------|--------|------|------|
| **标题区** | 0.3" | 1.5" | 1.2" | 大标题（仅一行或两行） |
| **副标题区** | 1.5" | 2.0" | 0.5" | 副标题/小标题（可选） |
| **内容区** | 2.0" | 5.8" | 3.8" | 正文/图片/数据 |
| **页脚区** | 6.0" | 7.5" | 1.5" | 签名/来源（仅封面/尾页） |

**⚠️ 严禁跨区域放置元素！**

#### 2.8.3 封面页布局规范

```python
# 封面页精确坐标模板

    "title": {
```

*   ❌ 图片高度超过 3.0"

*   ❌ 图片高度超过 4.0"

#### 2.8.7 内容溢出检测规则

**在生成代码时，必须进行以下检查：**

```python
def check_overflow(content_lines, max_height_inches, line_height_pt):
    """检测内容是否会溢出"""
    total_height_pt = len(content_lines) * line_height_pt
    total_height_inches = total_height_pt / 72
    if total_height_inches > max_height_inches:
        raise OverflowError(f"内容高度 {total_height_inches}\" 超过限制 {max_height_inches}\"")
```

**经验法则：**
*   每条 bullet (26pt) ≈ 0.5" 高度
*   内容区 4.5" 最多容纳 **8-9 条** 普通 bullet
*   强调文字 (60pt) ≈ 1.0" 高度
*   内容区最多容纳 **1 条** 强调文字 + **4-5 条** 普通 bullet

## 第三部分：页面类型标准 (Page Templates)

使用代码生成 PPT 时，必须将页面归类为以下五种标准类型之一：

| 类型 | 描述 | 布局规则 |
|------|------|----------|
| **封面页 (Cover)** | 标题+副标题+Hero图 | 图片居中，副标题在图片下方，签名在页脚 |
| **目录页 (Agenda)** | 大纲列表 | 纯文字，左对齐或居中 |
| **内容页 (Content)** | 纯文字列表 | 无图片，文字区域占满 |
| **数据页 (Data)** | 大数字+说明 | 数字超大居中，说明在下方 |
| **图文页 (SplitImage)** | 左文右图 | 左侧 50%，右侧 50%，中间 0.5" 间隙 |
| **尾页 (Closing)** | 谢谢+签名 | 居中大字，签名在页脚 |

## 第五部分：签名设计规范 (Signature Design)

**签名只出现在 封面页 和 尾页。中间页严禁出现任何作者信息。**

### 5.1 签名内容
*   **首页 (Cover)**: `天翼云湖北分公司 作者：王理`
*   **尾页 (Closing)**: `天翼云湖北分公司 王理`

### 5.2 签名样式 (醒目设计)
为了使签名醒目且美观，采用以下样式：
*   **位置**: 页面底部，居中。
*   **公司名**: 字号 18pt，颜色 `RGB(0, 191, 255)` (天蓝色，呼应天翼云品牌)。
*   **作者名**: 字号 22pt，颜色 `RGB(255, 215, 0)` (金色)，加粗。
*   **布局**: 公司名在上，作者名在下，形成两行居中的签名块。

### 5.3 代码实现示例
```python
def add_signature(slide, prs, is_cover=True):
    """仅在封面和尾页调用此函数"""
    SLIDE_WIDTH = prs.slide_width
    SLIDE_HEIGHT = prs.slide_height
    
    # 签名块位置：页面底部
    sig_width = Inches(5)
    sig_height = Inches(0.8)
    sig_left = (SLIDE_WIDTH - sig_width) / 2
    sig_top = SLIDE_HEIGHT - sig_height - Inches(0.2)
    
    box = slide.shapes.add_textbox(sig_left, sig_top, sig_width, sig_height)
    tf = box.text_frame
    
    # 第一行：公司名
    p1 = tf.add_paragraph()
    p1.text = "天翼云湖北分公司" + (" 作者：" if is_cover else " ")
    p1.font.size = Pt(18)
    p1.font.color.rgb = RGBColor(0, 191, 255)  # 天蓝色
    p1.alignment = PP_ALIGN.CENTER
    
    # 第二行：作者名 (金色加粗)
    p2 = tf.add_paragraph()
    p2.text = "王理"
    p2.font.size = Pt(24)
    p2.font.color.rgb = RGBColor(255, 215, 0)  # 金色
    p2.font.bold = True
    p2.alignment = PP_ALIGN.CENTER
```

## 第八部分：学术论文解构与通俗化 (Paper Deconstruction) ⭐ 核心能力

**当输入是学术论文时，必须执行"概念解构→雷军化→配图增强"的三步转化。**

### 8.1 论文阅读与核心概念提取

#### 8.1.1 必须识别的论文要素

| 要素类型 | 识别方法 | PPT 转化方式 |
|----------|----------|--------------|
| **核心创新点** | Abstract 中的 "we propose/introduce" | 作为 PPT 的核心主题，需重点解构 |
| **技术术语** | 首次出现时有定义或缩写的词汇 | 每个术语需单独解释页 |
| **关键数据** | 表格/图表中的对比数据 | 转化为数据页，大数字展示 |
| **对比对象** | "compared to/vs/baseline" | 转化为对比分析章节 |
| **局限性/未来工作** | Conclusion 中的 limitations | 转化为"诚实的挑战"（增加真诚感） |

#### 8.1.2 核心术语提取模板

```markdown
## 论文核心术语清单

| 术语 | 论文原文定义 | 通俗翻译 | 配图需求 |
|------|-------------|----------|----------|
| [术语1] | [原文定义] | [用大白话解释] | ✅/❌ |
| [术语2] | [原文定义] | [用大白话解释] | ✅/❌ |
```

### 8.2 概念解构方法论 (Deconstruction Methodology)

**论文中的每个晦涩概念，都必须经过"三层解构"才能用于演讲。**

#### 8.2.1 三层解构框架

```
原始概念 (论文语言)
    ↓
第一层：技术翻译 (准确但仍专业)
    ↓
第二层：比喻转化 (用日常场景类比)
    ↓
第三层：情感升华 (讲故事、讲意义)
```

#### 8.2.2 解构示例：N-gram Hashing

| 层级 | 解构结果 |
|------|----------|
| **原始** | "We use context-dependent N-gram hashing to generate deterministic indices for memory lookup, achieving O(1) complexity." |
| **技术翻译** | 把输入的词序列转成一个固定的数字编号，用这个编号直接查表，查询时间恒定。 |
| **比喻转化** | 就像字典给每个词一个页码。你说"北京"，我立刻算出页码是12345，直接翻到那页，不用从头找。 |
| **情感升华** | 朋友们，这就是算法的力量——用一个简单的数学技巧，把"大海捞针"变成了"开门取钥匙"。O(1)，一秒都不用等！ |

#### 8.2.3 解构示例：Conditional Memory

| 层级 | 解构结果 |
|------|----------|
| **原始** | "Engram introduces Conditional Memory as a complementary axis of sparsity to existing MoE architectures." |
| **技术翻译** | Engram 在 MoE（专家混合）之外，增加了一个新的"稀疏维度"——不仅选择用哪些神经元计算，还能选择查哪些记忆。 |
| **比喻转化** | MoE 是"选哪个专家来回答问题"，Engram 是"选哪本参考书来翻阅"。两者结合，既有专家动脑子，又有书本做参考。 |
| **情感升华** | 想象一下：天才学霸自己想答案，聪明学生翻参考书。Engram 让 AI 两者兼备——既能想，也能查。这不是替代，而是升级！ |

#### 8.2.4 解构示例：Zipf Distribution

| 层级 | 解构结果 |
|------|----------|
| **原始** | "Natural language N-grams follow Zipf's law, enabling a multi-level cache hierarchy." |
| **技术翻译** | 自然语言中，少数词语出现频率极高，大多数词语很少出现。这个规律叫 Zipf 定律，可以用来设计多级缓存。 |
| **比喻转化** | 就像你手机里的 App。微信、抖音每天用，放首屏。某个三年前下的 App 从来不用，藏在最后一屏。 |
| **情感升华** | 这就是工程的智慧——顺应自然规律，而不是逆天而行。热门的放手边，冷门的放仓库。Engram 这么做，既快又省！ |

### 8.3 雷军化表达转化规则 (Lei Jun Style Transformation)

**解构完成后，必须用雷军风格重新组织语言。**

#### 8.3.1 雷军化公式

```
雷军化表达 = 痛点共鸣 + 数据冲击 + 比喻故事 + 金句收尾
```

#### 8.3.2 雷军化模板

| 原始风格 | 雷军化改造 |
|----------|-----------|
| "该方法实现了 O(1) 复杂度" | "朋友们，O(1) 是什么概念？无论表有多大——100亿、1000亿——查询时间都是恒定的。一秒都不用等！" |
| "实验结果表明 BBH 提升 5.0 分" | "大家看这个数字——+5.0 分！这是什么概念？就像高考多考 5 分，可能从二本变一本。一个 27B 的小模型，打败了 160B 的巨无霸！" |
| "该架构降低了内存访问延迟" | "还记得地铁站的闸机吗？以前挤得过不去。现在？快速通道，刷脸就进！" |
| "我们提出了一种新的稀疏范式" | "朋友们，我们今天不是来改进的，我们是来革命的。这是一个全新的维度！" |

#### 8.3.3 必须应用的雷军元素

| 元素 | 应用位置 | 示例 |
|------|----------|------|
| **痛点开场** | 每个技术概念解释前 | "你们有没有遇到过这种情况：AI聊着聊着就忘了你说过的话？" |
| **数字具象化** | 每个关键数据 | "100B 参数是什么概念？相当于把整个维基百科记在脑子里。" |
| **对比故事** | 新旧方案对比 | "三年前需要一个机柜，今天只要一块卡。" |
| **真诚脆弱** | 讲到困难时 | "说实话，刚看到这个数字我也不敢相信。" |
| **金句收尾** | 每个章节结束 | "这，就是算法的力量——以小博大！" |

## 第九部分：动态图像生成 (Dynamic Visualization)

**对于 PPT 中特别重要的核心论点，可以生成动态图像（录屏/演示动画）进行更生动的解读。**

### 9.1 何时使用动态图像
| 重要性级别 | 判断标准 | 处理方式 |
|------------|----------|----------|
| **普通** | 常规信息点 | 纯文字或静态配图 |
| **重要** | 核心论点、关键数据 | 静态配图 + 大字强调 |
| **极其重要** | 颠覆性结论、核心对比、Demo 演示 | **动态图像** |

### 9.2 动态图像类型

| 类型 | 适用场景 | 生成方法 |
|------|----------|----------|
| **流程动画** | 数据流、推理过程 | 使用 `browser_subagent` 录制 HTML/CSS 动画页面 |
| **对比动画** | A vs B 切换演示 | 录制两张图交替显示的屏幕 |
| **交互演示** | 产品 Demo、代码运行 | 录制浏览器操作过程 |
| **数据可视化** | 图表增长、趋势变化 | 录制 D3.js/Chart.js 动态图表 |

### 9.3 动态图像生成流程

1.  **创建演示页面**：
    *   使用 HTML/CSS/JavaScript 创建一个极简的动画演示页面。
    *   动画应在页面加载后 1-2 秒内自动播放。
    *   总时长控制在 5-10 秒。

2.  **录制动画**：
    *   调用 `browser_subagent` 打开本地 HTML 文件。
    *   设置合适的录制名称（如 `kv_cache_flow_animation`）。
    *   等待动画播放完成。

3.  **输出文件**：
    *   录制结果自动保存为 `.webp` 格式。
    *   将路径记录供 PPT 引用（注意：python-pptx 不支持直接嵌入动图，需转为视频或提示用户手动插入）。

### 9.4 动态图像 HTML 模板示例

```html
<!DOCTYPE html>
<html>
<head>
  <style>
    body { background: #0f0f19; display: flex; justify-content: center; align-items: center; height: 100vh; margin: 0; }

    .block { width: 100px; height: 100px; background: linear-gradient(135deg, #00bfff, #ffd700); border-radius: 10px; animation: pulse 1s infinite; }
    @keyframes pulse { 0%, 100% { transform: scale(1); } 50% { transform: scale(1.2); } }
    h1 { color: white; font-family: Arial; margin-top: 20px; }
  </style>
</head>
<body>
  <div class="container">
    <div class="block"></div>
    <h1>数据流动中...</h1>
  </div>
</body>
</html>
```

### 9.5 动态图像使用注意事项
*   **PPT 兼容性**：PowerPoint 支持 `.mp4` 视频嵌入，但不支持 `.webp` 动图。建议：
    *   将动态内容转换为 `.mp4`。
    *   或在 PPT 中添加超链接指向外部文件。
    *   或提示用户在演示时切换到浏览器播放。
*   **文件大小**：控制动态图像在 5MB 以内，避免 PPT 过大。

## 第十部分：内容增强决策树 (Content Enhancement Decision Tree)

在生成每一页 PPT 时，按以下决策树判断是否需要增强：

```
开始
  │
  ├─ 页面包含抽象技术术语？
  │    ├─ 是 → 生成解释性配图 (§8)
  │    └─ 否 → 继续
  │
  ├─ 页面是核心论点/关键对比？
  │    ├─ 是 → 该点是否值得动态演示？
  │    │         ├─ 是 → 生成动态图像 (§9)
  │    │         └─ 否 → 使用数据页(大字强调)或图文页
  │    └─ 否 → 使用内容页(纯文字)
  │
  ├─ 页面是封面/尾页？
  │    ├─ 是 → 添加签名 (§5)
  │    └─ 否 → 不添加签名
  │
结束
```
